## How to Sign This CLA

### Individual Contributors

By submitting a Contribution, you automatically agree to this CLA. No separate signature is required, but we recommend:

1. **Read this Agreement carefully**
2. **Add your name to the contributors list** (optional): Create a PR adding yourself to `CONTRIBUTORS.md`
3. **Submit your Contribution** via pull request

Your first pull request will include a CLA bot that confirms your agreement.

### Corporate Contributors

If you are contributing on behalf of your employer, please:

1. **Download the Corporate CLA**: [CORPORATE_CLA.md](CORPORATE_CLA.md) (coming soon)
2. **Have an authorized signatory sign it**
3. **Email the signed CLA** to themailmaninbox@gmail.com
4. **Proceed with contributions** once confirmed

## Why Do We Need a CLA?

### Protects Everyone

- **Protects the Project**: Ensures we have clear rights to all code
- **Protects You**: Confirms you have the right to submit your code
- **Protects Users**: Ensures they can use the software without legal risk

### Enables Dual Licensing

The Project uses:
- **Business Source License (BSL)** for the open-source version
- **Commercial Licenses** for businesses that need proprietary use

The CLA allows us to offer both without legal complications.

### Allows Future License Changes

If we need to change licenses in the future (e.g., convert to Apache 2.0 after the Change Date), the CLA allows us to do so without tracking down every contributor for permission.

### Industry Standard

CLAs are used by major projects like Apache, Google, Microsoft, Linux Foundation, and many others. This is a well-established practice.
